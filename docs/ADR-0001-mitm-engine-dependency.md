# ADR-0001: MITM 引擎依赖策略

- 日期：2026-03-18
- 状态：Accepted

## 背景

项目主入口为 SOCKS5，目标能力包括 HTTPS MITM、规则分流、脚本改写、抓包导出。

## 决策

1. SOCKS5 主路径使用自研 MITM 数据面，不把 GoProxy 作为核心执行链路。
2. 通过 `MITMEngine` 抽象保留可插拔能力。
3. GoProxy 作为 HTTP 代理兼容模式的可选适配层，不直接 fork。
4. 仅当出现阻断性能力缺口时，才评估最小补丁 fork。

## 原因

1. GoProxy 主模型是 HTTP Proxy + CONNECT，与 SOCKS5 握手后原始 TCP/TLS 流语义不同。
2. 主路径需要深度控制分流、TLS 终结、脚本和抓包策略，避免协议层错配。
3. 使用抽象层可降低后续替换成本，避免业务代码绑定第三方细节。

## 后果

1. M1 交付会有一定自研工作量。
2. 长期可维护性更高，且更匹配项目目标能力。
